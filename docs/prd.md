# Tmux Markdown Mailer System (TMMS) 仕様書

## 1. システム概要
Tmux Markdown Mailer System (TMMS) は、tmuxのペイン（ターミナル）を仮想的なメールクライアントとして扱い、Markdownファイルを用いて非同期メッセージングを行うシステムである。主にAIエージェント間の自律的な通信や、人間とエージェント間のインターフェースとして機能する。
ファイルシステムをメッセージキューとして利用し、標準的なUNIXコマンドラインツールとの高い親和性を持つ。

## 2. アーキテクチャ構成
本システムは、バックグラウンドでルーティングと通知を行う「サーバ（デーモン）」と、メッセージを送信するための「クライアント（CLI）」で構成される。受信メッセージの閲覧や状態管理は、標準のファイル操作コマンド（`ls`, `cat`, `rm` 等）に委ねる（KISS原則）。

### 2.1. tmux ペイン・メタデータ
各tmuxペインは、システム参加のための設定をtmuxのユーザー定義オプション（`@変数名`）として保持する。

| 変数名 | 型 | 説明 | 必須 |
| :--- | :--- | :--- | :--- |
| `@tmms_enabled` | boolean | TMMSの対象とするかどうかのフラグ (`true`/`false`)。 | 必須 |
| `@tmms_name` | string | ペインの識別子。メールの送信元(`From`)および宛先(`To`)のアドレスとなる。 | 必須 |
| `@tmms_outbox` | string | 送信トレイとなるディレクトリの絶対パス。 | 必須 |
| `@tmms_inbox` | string | 受信トレイとなるディレクトリの絶対パス。 | 必須 |
| `@tmms_notify_cmd` | string | `inbox` にファイルが存在する際に実行される通知コマンド。 | 必須 |
| `@tmms_reply_to` | string | `tmms post` 実行時のデフォルトの宛先アドレス。 | 任意 |

### 2.2. メッセージフォーマット
メッセージはYAMLフロントマターを含むMarkdownファイル (`.md`) とする。
他のツールとの互換性を保つため、システムが管理するメタデータには `tmms_` プレフィックスを付与する。

**必須/システム予約キー:**
* `tmms_id`: メッセージの一意なID（UUID v4など）。
* `tmms_from`: 送信元の `@tmms_name`。
* `tmms_to`: 宛先の `@tmms_name`。
* `tmms_created_at`: 作成日時（ISO 8601形式。例: `2026-03-22T05:15:45Z`）。

**任意キー:**
* `tmms_subject`: メッセージの件名。
* `tmms_thread_id`: スレッドを追跡するためのID。

### 2.3. ファイル命名規則
視認性と時系列でのソートを容易にするため、生成されるファイル名は以下のフォーマットに従う。

* **フォーマット:** `{YYYYMMDD-HHMMSS}_{From}_to_{To}_{ShortID}.md`
    * `ShortID` は `tmms_id` (UUID) の先頭8文字を推奨。
* **例:** `20260322-143005_agent-alpha_to_user-main_a1b2c3d4.md`

## 3. サーバ仕様 (`tmms server`)

サーバプロセスは、設定された間隔でtmuxサーバ（起動中のセッション）とファイルシステムをポーリングし、ルーティングと通知を実行する。サーバ自身はtmuxセッション内で動作する必要はなく、tmuxバイナリが利用可能であればセッション外からも起動できる。

### 3.1. コマンドインターフェース
* `tmms server`: サーバプロセスをフォアグラウンド（またはデーモン）で起動する。`-c` オプションで設定ファイルを明示指定できる。省略時は以下の順序で設定ファイルを探索する:
    1. `~/.tmms.yml`
    2. `~/.tmms/config.yml`
    3. `~/.config/tmms/config.yml`
* `tmms server init`: デフォルトの設定ファイルを `~/.config/tmms/config.yml` に出力する。`-o` オプションで出力先を変更できる。出力先の親ディレクトリが存在しない場合は自動作成する。

### 3.2. サーバ設定ファイル
* `polling_interval`: ポーリングの実行間隔（秒）。デフォルト: `60`。
* `dead_letter_dir`: 宛先が存在しないメッセージを隔離するディレクトリの絶対パス。デフォルト: `~/.local/share/tmms/dead_letter`。
    * サーバ起動時に `dead_letter_dir` 内に `.md` ファイルが存在する場合、件数とパスを警告として stderr に出力する。

### 3.3. サーバプロセスの挙動 (毎ポーリング時)

サーバは以下の Task A と Task B を順に実行する。

#### Task A: メールのルーティング（送信処理）
1.  tmuxサーバ上の全セッションから `@tmms_enabled=true` が設定されている全ペインの情報を取得する。
2.  各ペインの `@tmms_outbox` ディレクトリ内の `.md` ファイルをスキャンする。
3.  ファイルのフロントマターから `tmms_to` の値を読み取る。
4.  `tmms_to` の値と一致する `@tmms_name` を持つペインを検索する。
    * **[成功時]** 宛先ペインの `@tmms_inbox` ディレクトリが存在しない場合は自動作成し、該当ファイルを移動 (`mv`) する。
    * **[失敗時 (宛先不明)]** `dead_letter_dir` が存在しない場合は自動作成し、該当ファイルを移動 (`mv`) して、エラーログを出力する。

#### Task B: 受信通知処理
1.  `@tmms_enabled=true` が設定されている全ペインの情報を取得する。
2.  各ペインの `@tmms_inbox` ディレクトリ内の `.md` ファイルをスキャンする。
3.  ファイルが存在する場合、各ファイルに対してペインに設定された `@tmms_notify_cmd` を評価・実行する。
4.  **変数展開:** 実行前、コマンド文字列内の `${{変数名}}` を対象ファイルの情報に置換する。
    * `${{filepath}}`: 対象ファイルの絶対パス
    * `${{tmms_id}}`, `${{tmms_from}}`, `${{tmms_to}}`, `${{tmms_subject}}`, `${{tmms_thread_id}}`: フロントマターの値
5.  **※状態管理に関する留意点:** ファイルが `inbox` 内に存在する限り、次回のポーリング時にも通知コマンドが再実行される。メッセージの処理完了とファイルの削除（または別ディレクトリへの退避）は、受信側（クライアント/エージェント）の責任とする。

## 4. クライアント仕様 (`tmms post`)

`tmms post` コマンドは、現在のtmuxペインのコンテキストを読み取り、適切なメタデータを付与してメッセージを `outbox` に配置する。

### 4.1. コマンドインターフェース
```bash
tmms post [OPTIONS] [FILE]
# または標準入力から
echo "Body text" | tmms post [OPTIONS]
```

### 4.2. オプション
* `-t, --to <ADDRESS>`: 宛先アドレス (`@tmms_name`)。必須（指定がない場合は `@tmms_reply_to` を使用。どちらもなければエラー）。
* `-s, --subject <TEXT>`: メールの件名（任意）。
* `-f, --file <PATH>`: 送信するMarkdownファイルのパス。

### 4.3. 実行フローとフロントマター処理ロジック
1.  **コンテキスト検証:** 現在のターミナルがtmuxペイン内であることを確認し、`@tmms_name` および `@tmms_outbox` を取得する。不足している場合はエラー終了。
2.  **宛先解決:** `-t` オプションの値を優先し、指定がなければtmuxの `@tmms_reply_to` の値を使用する。
3.  **入力読込:** 指定されたファイル、または標準入力からテキストを読み込む。
4.  **メタデータ・マージ処理:**
    * 読み込んだテキストの先頭が `---` で始まるYAMLフロントマターを持つ場合、それをパースする。
    * `tmms_` プレフィックスを持たない既存のキー（例: `tags`, `aliases`）は **すべて保持** する。
    * システムが必要とする `tmms_` キー（`id`, `from`, `to`, `created_at` 等）を生成・適用する。入力側に同名の `tmms_` キーが既に存在した場合は、**システムが生成・指定した値で上書き** する。
5.  **出力:**
    * マージ済みのフロントマターと本文を結合する。
    * `@tmms_outbox` ディレクトリが存在しない場合は自動作成し、規定のファイル命名規則に従ってファイルを書き出す。
    * 入力元がファイル（`-f` または引数）であった場合も元のファイルは削除しない（コピーとして outbox に配置する）。標準入力の場合は新規作成のみ。
